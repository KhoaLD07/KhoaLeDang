COBALT STRIKE

TOPIC: PLATFORM COMPONENTS, SETUP, COMMANDS, BOF, CNA

Cobalt Strike is not just a single tool but a platform consisting 4 core components:

  • Team Server: The center command server. It listens for incoming connections from beacon, stores all the stolen data, manage all the clients and serves payload to beacon on victim machine.
  • Client: Will connect to the teamserver. Many clients can connect to the server and interact with each other on real time.For example: One person can Elevate Privileges, The others can extract information.
  • Beacon: The core payload executed on the victim computer. It only operates asychronoulsy, it sleeps for a period of time, wakes up contact with the team server to see if there's any new task. If there's nothing it will continue to sleep. This makes it harder to spot it. Beacon can transfer through many route: HTTP/HTTPS, DNS, SMB.
  • Malleable C2: Firewalls nowadays are very smart it blocks a .exe spotanously send encrypted data through a suspicious IP address. Malleable helps create configs, these configs will put the data inside fake HTML tags, So it will look like just a normal connection.

Beacon Object File

Beacon Object File (BOF): BOF is a compiled C program that hasn't been linked (.o or .obj). Normal compilation links object files with system libraries to create a .exe file. BOF skips the second step, it only needs the .o file to run. Instead of relying on Windows, the Beacon will acts as the linker in the memory.

- Life span of a BOF. When you execute by inline-execute, Client Cobalt Strike will read .obj from the client. Encrypt it and then send the BOF through the server right into the RAM of victim machine.
After that Beacon receive the BOF, allocate a hidden memory (read write), then put the code inside.
with the DECLSPEC_IMPORT, Beacon will search inside Windows to find the address of the function and give it to BOF.
When all done Beacon will add the execute Privilege for the memory and call the function go(). Finally when go() is finished Beacon will free the memory, clean everything.

Why don't we use PowerShell and C# instead

PowerShell is heavily monitored by AMSI (Antimalware Scan Interface). which scans and logs everything you scripts executed.
C# (execute-assembly): This technique force beacon to create a sacrifical process (notepad or werfault.exe) to inject C# code inside. The creation of a new process is easy to be spotted.
BOF is Unmanaged Code that runs inline meaning it's not creating any new process.

Weakness: crash together.

- Because the memory is inside Beacon itself so if your BOF get buffer overflow, Null Pointer Deference, Forgot to allocate memory,...
The Beacon will crash immediately and lost connection to the victim machine.
We need to use BOF because beacon will use BOF to inject straight into the victim RAM.
It also doesn't create a process, it has a small size so it can be transfer through internet very quickly

To get the args from the command line we need to use BeaconDataParse to parse the data to datap struct. We need to extract the data from buf, len.

    target_dir = BeaconDataExtract(&data, &length);

BeaconDataExtract will extract String from the data to target_dir. BeaconDataInt will extract Integer from the data. The order depends on how we bof_pack the arguments in cna.

    int out_size = 0;
    BeaconFormatAlloc(&out_buffer, 8192);

If we use BeaconPrintf it will sent the a lot data to the teamserver simutaneously. That's easy to be detected. So we need to use BeaconFormatAlloc to allocate a buffer to store the data. Then put all the print data inside the buffer by using BeaconFormatPrintf. Finally use BeaconFormatToString to convert the buffer to a string and print it to the teamserver. This method only send one packet to the teamserver.

CNA: Aggressor Script intergrated inside COBALT STRIKE using Sleep language

#$0 is the source command. eg: "-ls C:\Windows"
#$1 BeaconID: is beacon ID
#$2 the first argument "C:\Windows"
#$3 #$4 is the second, third... 

alias -ls {
    local('$barch $handle $data $args');
}

declarations: $ is a variable and @ is an array.

if (size(@_) != 2) {
    berror($1, "How to use: -ls [path]");
    return;
}

Checks if input is valid.

$barch  = barch($1);
$handle = openf(script_resource("BOF2. $+ $barch $+ .obj"));
$data   = readb($handle, -1);
closef($handle);
  • $barch: checks victim machine architecture (x86 or x64)
  • $handle = openf(...): open file using the arch
  • $data = readb(...): read whole file into $data (-1 = all)
  • closef($handle): close the handle
$args = bof_pack($1, "z", $2);
  • "Z" is Wide String (UTF-16). Very important because many Win32 APIs require wide string (e.g. L"C:\\Windows"). Extracts to: wchar_t* wstr = (wchar_t*)BeaconDataExtract(&parser, &len)
  • z = string, i = integer, s = short, b = binary. You will need to provide an empty int to catch the string length.
  • bof_pack will take arguments from your command line but you need to write the type of the arguments in order.
    Example: bof_pack($1, "iz", $2, $3) means it will take the first argument as an integer, and the second as a string.
  • IMPORTANT: you have to take the arguments from the pack in the same order too. In this example you need to take BeaconDataInt() then BeaconDataExtract().
btask($1, "Sending to Beacon...");
beacon_inline_execute($1, $data, "go", $args);
  • beacon_inline_execute will execute the payload
  • $1 is the ID of the beacon
  • $data is our BOF.obj file
  • go is the entry point
  • $args is the arguments just loaded, will get pushed into char* args inside go() function
  • OPSEC RULE: Never ask for RWX (Read-Write-Execute) at the same time. EDRs will catch you. Always allocate RW (Read-Write) → Copy code → VirtualProtect to RX (Read-Execute) → Run.
